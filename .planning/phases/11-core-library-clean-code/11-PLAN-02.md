### Task 11.2-B: Simplify Priority 1 Functions (Complexity > 15)

<read_first>
- .planning/phases/11-core-library-clean-code/reports/11.2-COMPLEXITY.md (Priority 1 list)
- The actual source files containing Priority 1 functions
- bin/lib/decorators/LogExecution.ts (for decorator usage)
</read_first>

<action>
1. For each Priority 1 function (complexity > 15 OR length > 100 lines):

2. Apply guard clauses to eliminate deep nesting:
   - Replace pattern: `if (condition) { /* main logic */ }` with `if (!condition) return defaultValue;`
   - Replace pattern: `if (!x) { throw error; } else { /* logic */ }` with `if (!x) throw error; /* logic */`
   - Target: Reduce nesting from 4+ levels to max 3 levels

3. Extract helper methods for sub-tasks:
   - Identify logical blocks within the function (validation, transformation, persistence, etc.)
   - Extract each block to a separate private method with descriptive name
   - Example: Extract `validateInput()` , `transformData()`, `persistResult()`
   - Each extracted method should be < 30 lines

4. Replace complex conditionals with boolean methods:
   - Replace: `if (x > 0 && x < 100 && y !== null && y !== undefined)`
   - With: `if (this.isValidRange(x, y))`
   - Extract complex boolean expressions to private methods with descriptive names

5. Apply @LogExecution decorator to all extracted methods

6. Update tests if function signatures change

7. Document each refactoring in commit messages
</action>

### Task 11.2-C: Simplify Priority 2 Functions (Complexity 12-15)

<read_first>

- The actual source files containing Priority 2 functions
- Examples from Task 11.2-B refactoring
</read_first>

<action>
1. For each Priority 2 function (complexity 12-15 OR length 75-100 lines):

2. Apply the same simplification techniques as Task 11.2-B:
   - Guard clauses for early returns
   - Extract helper methods for sub-tasks
   - Replace complex conditionals with boolean methods

3. Use table-driven methods for repetitive conditionals:
   - Replace switch statements with lookup tables where appropriate
   - Example pattern:
     ```typescript
     // BEFORE: Large switch statement
     switch (type) {
       case 'A': return handleA();
       case 'B': return handleB();
       case 'C': return handleC();
     }

     // AFTER: Table-driven
     const handlers = { 'A': handleA, 'B': handleB, 'C': handleC };
     return handlers[type]();
     ```

4. Use strategy pattern for complex algorithmic conditionals:
   - If function has multiple algorithm branches based on type/mode
   - Create strategy interface and concrete implementations
   - Delegate to strategy from simplified main function

5. Apply @LogExecution decorator to all extracted methods

6. Update tests if needed
</action>

- All Priority 2 functions have complexity < 10 after refactoring
